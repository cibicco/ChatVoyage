## 2026-07-11 アルバム閲覧体験とデータ基盤の製品化

### 背景/目的
Chat Voyage の画像数とアルバム数が増え、従来の `index.html` / `albums.html` は「情報を確認する画面」にはなっていたが、「画像を気持ちよく見て、比較し、次の改善判断へ進む画面」としては弱かった。今回の一連の作業では、キャラクターアルバムと通常日次アルバムを明確に分けつつ、全体を横断して見られる導線、画像中心の一覧、アルバム内の前後移動、評価メモ、メタデータの正規化までを整え、Chat Voyage を静的な画像置き場からレビュー可能な創作アーカイブへ近づけることを目的にした。

### 成果
- アルバムと画像のソースを `data/albums/**/*.json` に寄せ、`assets/album-data.js`、`index.html`、`albums.html`、`album.html` を生成物として扱う構造へ整理した。画像ごとに `width`、`height`、`aspectRatio` を持たせ、アルバムごとに `preferredAspectRatio` を持つ方針を明文化した。
- 旧 `assets/daily/` と直下 `notes/` に散らばっていた画像・メモを、`assets/albums/daily/YYYY/MM/`、`assets/albums/characters/shino/YYYY/MM/`、`notes/albums/...` へ移し、キャラクターと通常日次の保管場所を分けた。
- `albums.html` は、キャラクター/日常/すべてのモードを持つアルバムブラウザにした。`すべて` では分類見出しで分断せず、キャラクターと日常を混在表示する。並び順は「アルバム日付 新しい順/古い順」「日付ごと」など、アルバムの見方に合わせた表現へ改めた。
- `index.html` は、画像一覧として日常/キャラクター/すべてを横断できるようにした。カードのキャプションは重複した画像タイトルではなく、`日付 / 都市 / 何枚目` と `カテゴリ / 具体ラベル / 絵柄` の2段構成へ整理した。
- `album.html` は、画像を大きく見るためのビューとして、左右キー・クリック・ボタンによる画像移動、ライトボックス、好みフィードバック、アルバムセレクト、前後アルバム移動を備えた。メモや全フィードバック書き出しも、一覧より詳細画面中心の導線へ寄せた。
- `旧カテゴリ` 依存を弱め、`occasion`、`venue`、`activity`、`outfit`、`locationDetail` を使って、場所・場面・行動・服装を別軸で評価できるようにした。
- `daily-fashion-sketch` と `shino-album-generation` の役割を確認し、通常日次生成と紫乃固定キャラクター生成は既に分けて運用できる状態であることを確認した。今回のアルバムUIレビューについては、新規skill化よりも、まず `docs/album-json-contract.md` とhandoverに実装知識を残す段階と判断した。

### 課題/リスク
- Safari 実画面での確認は、セッション途中でブラウザ自動操作が `localhost:8765` に対して拒否された場面があり、最終的な画面崩れ確認はHTML/CSS/JSと静的検証中心になった。GitHub Pages本番とiPhone実機での確認はまだ残る。
- アルバムUI改善は進んだが、画像閲覧の理想形はまだ完成ではない。特に画像サイズ、余白、サムネイルの切り抜き、メタデータの量、モバイルでの操作密度は、実画面レビューを続けて詰める必要がある。
- 旧パスから新パスへの移行で削除扱いになったファイルが多い。表示参照と検証は通っているが、生成資産は project materials なので、今後も削除・退避・アーカイブの判断は慎重に扱う。
- スキル面では、通常生成と紫乃生成のskillは整ったが、アルバムUI/データ移行/Pages確認の繰り返しが増えるなら、将来的に `chat-voyage-album-ops` のような専用skill化を検討する余地がある。

### 検証

- `node --check assets/album-browser.js`
- `node --check assets/album-page.js`
- `python3 -m py_compile scripts/build_album_catalog.py`
- `git diff --check`

## 2026-07-04 紫乃アルバム制作基盤の具体化

### 背景/目的
Chat Voyage の紫乃アルバムを、単発の絵作りではなく、同一人物の生活・仕事・街・季節を継続的に描ける創作基盤にすることを目的にした。特に、25歳の成人女性としての一貫性、架空の港町の日常、平日と週末の差、金曜夜だけ表れる意外性を、生成・レビュー・保存の運用へ落とし込んだ。

### 成果
- 紫乃を「海沿いの温暖な架空港町で暮らす25歳の製品規格ラボのテストエンジニア」として再定義し、性格、生活圏、自宅、仕事、休日、金曜夜のDJ手伝い、水族館嗜好、服装の好みを整理した。
- 南紀/伊勢の気候感、ニュルンベルク風の旧市街、港町、水族館、路面電車、古い建物、規格ラボを組み合わせた舞台設定を作り、具体的な場面へ展開できる状態にした。
- `2026-06-06-13-shino-ise-weather-lab`、`2026-06-14-18-shino-ise-weather-lab`、`2026-06-19-shino-birthday-dj-night`、`2026-06-21-shino-coastal-lab-aquarium`、`2026-06-21-shino-weekend-port-town` など、紫乃の固定キャラクター日常アルバムを WebP 画像、notes、album catalog、index、generation log に接続した。
- 人物一貫性レビューを通じて、現在の25歳紫乃として強い候補、弱い候補、参照昇格候補を分ける運用を確認した。特に `2026-06-14-18` の `03`、`08`、`09` は今後の current-growth 参照候補として有力。
- 金曜夜の誕生日DJセットでは、紫乃を派手なパーティー人物にせず、低音、照明、紙フライヤー、機材、同僚の小さな祝い、帰り道、自室の余韻として描く方向を確立した。

### 課題
- 紫乃の衣装説明は黒/ネイビーへ寄りやすく、同一人物性を暗色服に依存しすぎるリスクがある。今後は顔、視線、行動、魚/ラボ/水族館/音楽/街の小物で一貫性を支える必要がある。
- 一部セットには WebP 変換後の PNG コピーが残っている。表示参照は WebP だが、生成物をプロジェクト材料として扱う方針上、削除には明示判断が必要。
- `2026-06-02-shino-mexico-city-mamey-coral` は `index.html`、`assets/album-data.js`、`albums.html` には同期済みだが、PNGコピーが残っているため、原本保護方針とWebP運用の境界を明示判断する必要がある。
- 既存の `daily-fashion-sketch` は4枚の日次生成を主対象にしており、紫乃の長尺固定キャラクターアルバム、i2i参照、参照昇格判断、異常出力時の停止判断は専用手順として切り出す余地がある。

### 失敗
- 一部の試行では、画像生成ツールが意図しない図解・インフォグラフィック風の出力へ逸れる挙動があった。以後は長尺セット生成前に1枚の短いテストを行い、人物画像として正常に返ることを確認してから続行する。
- 初期の引き継ぎメモには「6/19金曜夜セットは保存していない」という古い状態が混在していた。現在の作業ツリーでは同セットは6枚のWebP、notes、album data、index、logに存在しているため、文書化では作業ツリーの実状態を正とした。

### 心情
紫乃は、単に見た目を固定する段階から、街の中で何を見て、何に立ち止まり、どんな仕事と余白を持つ人物かを描ける段階に進んだ。一方で、固定キャラクターは少しの癖がすぐ反復になるので、今後は「紫乃らしさ」と「毎回違う日であること」を同時に見る運用が重要になる。

## 2026-07-04 Chat Voyage 日次生成とアルバム閲覧基盤の再整理

### 背景/目的
Chat Voyage を、単に日付ごとに画像を増やすギャラリーではなく、都市・具体的な場所・季節・人物像・服装・生活動作・閲覧体験を継続的にレビューできる創作アーカイブにすることを目的にした。画像生成の安定化だけでなく、似た絵が続く問題、アルバムUIの使い勝手、公開先、固定キャラクター混入の扱いまでを一体で見直した。

### 成果
- 日次生成の判断軸を、露出の有無ではなく、都市、季節、天気、時間帯、場所、活動、本人の快適さ、服の構造に基づく自然さへ寄せた。首元や肌見せを一律に抑えるのではなく、その場にいる成人女性の服装として成立するかを確認する方針にした。
- 年齢帯ごとの差分を、ラッキーカラーのトーンだけでなく、シルエット、生活シーン、素材、アクセサリ、靴、レッグウェア、態度まで広げた。
- 似た印象が続く原因として、服装だけでなく、market/gallery/lounge/transit の型、フェリー窓構図、同じような視線、短丈シアーソックス、glossy anime調、フルボディ正面寄りのファッション確認構図が反復していることを整理し、プロンプト/guardrailへ反映した。
- Notion は画像閲覧・レビューの本筋から外し、GitHub Pages 上の静的アルバムを主経路にした。既存の WebP、`index.html`、`albums.html`、`album.html`、CSS/JS、localStorage feedback を活かす方向に整理した。
- アルバムUIは、一覧、フィルタ、grid/list、ライトボックス、前後送り、メタ情報、好みフィードバックを持つプロダクトとして見直した。カテゴリに scene / clothing / place が混ざる問題に対して、`data-category` に加えて `occasion`、`venue`、`activity`、`outfit` を持たせる方向へ進めた。
- 6/18以降の複数日次セット、6/29 Lagos セット、WebP運用、アルバムredirect、notes/log/index/catalog接続を進め、生成後のレビューで受け入れ/再生成理由を明文化する運用を強めた。
- `daily-fashion-sketch` と `docs/daily-generation-workflow.md` の PNG 削除方針を、AGENTS の原本保護方針に合わせ、明示承認なしに生成元/PNGを削除しない表現へ修正した。

### 課題
- 作業ツリーは多数の画像・notes・album redirect・UI変更を含む dirty state で、まだひとまとまりの commit-ready 状態ではない。
- `2026-06-02-shino-mexico-city-mamey-coral` は catalog 同期と gallery validation は通っているが、固定キャラクター群を通常日次と同じ閲覧面に置くかはまだ設計判断が必要。
- 固定キャラクター群を通常日次と同じ場所に置くか、ChatVoyageIntime へ分けるか、`data-collection="character"` として Chat Voyage 内で明確に分離するかは、まだ最終整理が必要。
- GitHub Pages は閲覧経路として有力だが、公開状態、リポジトリ公開範囲、Pages反映、iPhone閲覧、UI崩れの再確認が必要。

### 失敗
- Notion は一度アップロード/DB設計まで進めたが、現状の画像レビュー体験としては重く、Chat Voyage の静的アルバムUIを置き換える主経路にはしなかった。
- 画像生成では、意図した人物画像ではなく図解風の出力、同一人物に見える出力、似たanime仕上げ、短丈シアーソックス偏重などが起きた。これらは単発の失敗ではなく、プロンプトが成功パターンへ戻る構造的な問題として扱う必要がある。
- 6/2 Shino Mexico City セットの扱いについて、当初は catalog 未同期と見ていたが、現状確認では `validate_gallery.py` は `errors: 0` だった。問題は validation 失敗ではなく、固定キャラクターと通常日次の設計上の分離である。

### 心情
今回の進捗は、画像を増やす作業から、何を良い画像と見なすか、どこで見るか、どう反復を避けるかまで含む運用品質の整理に移った。まだ散らかりは残っているが、Chat Voyage が「日々の女性の生活スナップとファッションを街ごとに眺めるアーカイブ」へ近づいた手応えはある。
